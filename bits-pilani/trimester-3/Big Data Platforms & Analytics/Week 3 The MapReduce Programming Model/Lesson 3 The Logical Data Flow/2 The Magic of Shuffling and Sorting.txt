Partitioning is the process of deciding
which reducer nodes will receive which
key. The network uses a simple but
powerful mathematical formula that's a
hash formula. So if you do a hash of
key modulo, the number of reducer,
imagine you have 3 reducer nodes node A,
node B, node C. When a mapper produces
the key index.stML, the framework
hashes that string into a number and
divides it by 3. If the result is 0, it
goes to node A. If the result is 1, it
goes to B. If this ensures that every
single instance of index.htML from
across the entire cluster is sent to
the exact same destination. Without
partitioning, the reducer would only
have a partial view of the truth. Once
the data arrives at the reducer node,
we move to the second part, which is
sorting. The framework doesn't dump the
data in a pile, it groups all the
values by their key. This is why it is
called shuffle and sort. By sorting the
data, the framework ensures that when
the reducer starts its work, all the
ones from index.html are lined up
together in a single list. This makes
the actual reduce function incredibly
efficient. The reducer doesn't have to
search through a messy database. It
just reads a perfectly organised stream
of values, one key at a time. Now look

Because so much data is moving across
