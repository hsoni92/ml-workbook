Now we can define a feature store. A
feature store is central system that
lets you define features once in code
or configuration. materialise those
features into both an offline store for
training and batch scoring, and an
online store for low latency survey.
And exposes them through API and a
catalogue or a user interface.
The core promise is simple, define a
feature once, use it consistently
offline and online, and avoid training
serving skew.
Let's break down the responsibilities
of a feature, typical feature store.
First, feature definition. You define
feature in code or configuration.
Include the entity key, time stamps,
and transformation logic. Second,
offline materialisation. The feature
store runs pipeline to compute feature
table in your data lake or warehouse.
And helps you do point in time, correct
joints for training. Third, online
materialisation. It pushes the latest
feature value into a low latency key
value store and keeps them fresh via
streaming or scheduled jobs. 4th,
serving API and catalogue, get
historical features for training. Get
online features for serving. And a
searchable registry so team can
discover and reuse existing features.
All of this is aimed at making feature
engineering reliable, reusable and
consistent.
