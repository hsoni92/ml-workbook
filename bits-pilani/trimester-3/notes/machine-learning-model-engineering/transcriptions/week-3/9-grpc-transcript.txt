Now let's talk about GRPC. Which is
another major way of exposing model
APIs. Specifically. In larger or more
performance sensitive system, GRPC
stands for Google Remote Processor Call.
It's a high performance contract first
framework that uses protocol buffers or
proto files instead of JSOs. This means
your request response formats are
defined in a strongly typed schema.
And the client and server both generate
code for that same contract. There are
a few advantages here because GRPC uses
this binary encoding. The payload are
much smaller and serialisation is much
faster compared to Jason. This makes
GRPC extremely efficient and high
throughput or low latency environments.
It also provides strong typing, so if
someone changes a field in the API, you
catch those issues in compile time
instead of discovering it in runtime.
On the right, you can see an example of
protofile. It defines predict request
message, a predict response message,
and a predict RPC method.
From this one file, GRPC can generate
clients for Python, Java, Go, C++, and
many other languages.
All using exact same contracts.
Practically, GRPC is most commonly used
in internal microservices, where many
services need to talk to each other
efficiently.
You will also see it in real-time
systems such as recommendation engine
or large distributed architecture,
where performance and strict structure
really matters.
So while rest is a great starting point,
GRPC becomes very valuable once you
scale up or need more speed and
structure.
So how do you choose between rest and
GRPC? Rest.
It is a simple, human friendly, works
almost everywhere, browser, mobile
devices, scripts, partner integrations,
everywhere.
It is often the best choice for
external clients, public APIs and
situations where ease of integration
matters more than raw performance. GRPC
gives you strong contracts via proto
definition.
It offers better performance and type
safety, especially for service to
service communication. It is often the
best choice for internal service at a
scale where you control both sides of
the communication. A common service
pattern in large system is uses rest as
the edge between internet and your back
end and uses GRPC inside the backend
between internal services, including
models serving microservices. In this
course, we will stick with RST plus
Jason for clarity, but it's useful to
know where GRPC actually fits.
