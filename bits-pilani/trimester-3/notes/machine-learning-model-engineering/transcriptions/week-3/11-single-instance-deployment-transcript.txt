We'll start with the simplest possible
deployment, one instance. In early
stages, we might have a single VM or a
single container running your model API.
It listens to the port like 8000 either
directly or behind a simple payload
balancer or a reverse proxy.
Configurations easily usually happen
through. Environment variables, model
parts, port, feature flags, secrets, or
small config files in JSON or YAML file.
For small internal tools or an early
prototype, this is good enough. It's
easy to reason about and easy to debug.
One process, one URL, minimal moving
parts.
The labs in this module will
essentially use this pattern, just
running your model API inside a Docker
container on localhost.
Let's connect this to typical build and
deploy pipeline. Step one, build. You
commit code to a repository. A CI
system, which is a continuous
integration system, runs tests. Linters
and checks to make sure things are
healthy. Step 2, package. You build a
docker image that contains your code.
Your model artefact, and all the
dependencies, you tag that image with
something like model service version
1.0.0. Step three, run.
You run that image as a container on a
VM in Kubernetes or in some container
platform. You pass in environment
variable, configure ports, and set up
any required secret.
This build package run flow is exactly
what you will practise in M3 labs, with
a fast API and docker.
Now let's talk about rollout. We ship
new version safely. In a toy setup, you
might just build a new docker image,
deploy it, and basically hope for the
best. In reality, shipping a new model
can introduce risk, bugs in the code or
configuration, a model that performs
worse on live data than in your offline
metrics. Latency regressions may be.
The new model is slower or more
resource hungry. Rollout strategies
exist to reduce the blast radius of a
bad release and let you test in
production with traffic, real traffic,
but in a controlled way.
