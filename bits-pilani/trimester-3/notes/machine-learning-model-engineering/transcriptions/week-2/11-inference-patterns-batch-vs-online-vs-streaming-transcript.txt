Streaming inference is what happens
when data arrives as a continuous flow
of events, not as a static file and not
as one of request. The model processes
those events as they happen. And the
output feeds into the other system.
Like alerts, dashboards, or downstream
services.
This is different from batch, where we
grab a big static data set, run a job,
and then we're done until the next run.
Online, where a caller makes a specific
request and expects a direct response.
With streaming, the mental model is a
long running pipeline. Events come in.
They flow through several stages, the
model is just one of those stages,
continuously transforming incoming
events into predictions or signals. Let
me give you a high level picture of
what a streaming pipeline looks like.
Conceptually, you have a few building
blocks, an event source. This is where
events originate. Applications, sensors,
devices, logs, click streams, a stream
transport, something that carries out
events like Kafka, Nisis, PUSub, or a
similar. Que stream system, a stream
processing layer. This is where we
define the logic of transforming events,
joining streams, aggregating, and so on.
Tools like Flink, spark streaming, or
custom
consumer live here, a model step.
Within that processing layer, we call
the model on each event or on small
mini batches of events. One or more
sinks. Where we send the outputs, maybe
to an alert system, a database, a
feature store, or a monitor dashboard.
The key idea events are constantly
flowing through the pipeline, and the
model is embedded as a stage in that
flow, rather than being triggered by
discrete batch job or single API
request. Let's contrast streaming. With
the two pattern you have already seen,
batch.
You start with a big static dataset.
You run one job, or rather one big job,

You get an individual request, you
