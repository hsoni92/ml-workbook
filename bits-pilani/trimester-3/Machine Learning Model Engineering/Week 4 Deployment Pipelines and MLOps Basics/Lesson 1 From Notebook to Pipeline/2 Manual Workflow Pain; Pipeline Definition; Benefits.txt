This is the core problem. Notebooks are

We introduce a deployment pipeline. A
deployment pipeline is essentially a
sequence of automated steps that takes
you from raw data to a trained model to
a deployed artefact or service. Instead
of relying on human to remember and
rerun each step, the pipeline encodes
it. The typical steps are. Data
preparation, train, evaluate, package,
and deploy. The key idea is that the
pipeline is the recipe for how we move
from one or an idea in a notebook to
something that's actually running in
production.
Let's zoom in on how pipeline stages
works.
Each stage in the pipeline.
Consume artefact.
Things like raw or clean data, configs,
or outputs from previous steps. And
produces artefacts. Clean data, trend
model, metrics, reports, containers,
and so on.
For example, data preparation step
loads raw data and output a cleaned
version data set. The train step,
consume the data step data sets, plus
configurations, and output a model
artefact and training metrics. The
evaluate step, consume the model and
the validation sets, and output
evaluation metrics, and maybe the
report. The package step consumes the
model and wrap it into a docker image
or model bundle. The deploy step Takes
the package and turns it into a running
service, and registered or a registered
model in a registry. Thinking in terms
of input and output.
Is actually important. It's what
actually allows us to track artefacts,
connect the steps, and rerun parts of
the pipeline when needed.
Here's a concrete, simple end to end
example. Step one, data preparation.
You load raw data from your source.
Clean it! Engineer features. And split
to train, validation and test, the
output a versioned clean data set. Step
2, train.
Use the data set plus a configuration
of hyperparameters to train a model.
The output, a model artefact, says a
.bKL file or saved weight. And training
matrices like loss curve.
Step 3, evaluate. Evaluate on
validation or test set. Reproduce
metrics like AUC or accuracy, and maybe
a smaller evaluation reports with plots.
Step 4, package and deploy. Take the
model artefact, bundle it into docker
image or some kind of deployable unit.
Push it to registry and deploy it as a
service. Register it as the latest
production model. This is just one
example, but the pattern is always the
same, clearly defined steps with clear
input and output chained together.
So why bother building pipeline instead
of just using notebook and manual
scripts? First, repeatability. If you
run into, run the same pipeline with
the same input, you get same outputs.
What makes experiment and debugging
much easier?
Second, audibility. You have a clear
record of what ran, when, and with what
artefact. That's critical for debugging
issues and for compliance or audit.
Third, speed. Once the pipeline is in
place, going from idea to production is
faster because you're not manually
stitching together each step.
At each time.
And 4th, fewer human errors.
The less we rely on copy paste and
manual file copying. The fewer oops, I
forgot to update the conflict movement
we have.
In short, pipeline make machine
learning more repeatable, traceable and
efficient.
