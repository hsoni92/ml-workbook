Let's start with a crisp definition.
Batch inference means. You take a large
set of input, thousands, millions,
sometimes billions of rows.
And run the model over all of them in
one job. And you write the predictions
somewhere, usually a database table, a
warehouse, or a file.
Time wise, this job runs periodically.
Maybe once a day, once an hour, or
every 15 minutes. And importantly,
There is usually no human sitting there
waiting on each individual prediction.
You care that the whole job finishes by
a certain time, but you don't care if
one specific row took 50 milliseconds
or 500 milliseconds to process, as long
as the entire batch completes on
schedule.
Here is what a batch inference job
typically looks like, end to end. First,
we read input data.
This might be a big CSV or perquet file.
Or a table in a data warehouse, or a
snapshot from a production database.
Then, we preprocess and featureize each
row. Apply the same feature engineering
we used at training time. Handle
missing values, encodings,
normalizations and so on. Next, we run
the model over all rows, sometimes row
by row, more often in chunk or mini
batches. And often in parallel across
multiple courses or machines. Finally,
we write predictions out, back into
database, or to a file like
predictions.csv. Or into a feature
store for later use by online systems.
The job has a clear start and a clear
end. We care a lot about how long did
this entire job take, did it finish
before our operational deadline?
