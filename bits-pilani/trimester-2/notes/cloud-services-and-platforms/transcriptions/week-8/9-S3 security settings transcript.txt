S3 security settings transcript

In this topic, we are going to talk about public access in S3 and more importantly, how
AWS helps prevent accidental data exposure using S3 block public access. This is a
critical topic because many real world cloud security incidents happen due to
misconfigured S3 buckets. Let's start with what public access actually means.

When an S3 bucket or object is public, it means anyone on the internet can access it.
No AWS login is required and the data can usually be accessed using a simple URL.
Now, public access is not always bad.

Some use cases like hosting a public website or sharing open documentation require
these things. So the real problem is accidental public access, which means buckets
often become public because of open bucket policies or public access control lists.
Overly permissive IAM policies or because block public access was disabled without
realising the impact.

This is where S3 block public access becomes extremely important. Think of S3 block
public access as a safety guardrail. Even if someone makes a mistake in a policy or
at ACL, block public access can stop the bucket from becoming public, which means
an additional layer of security to your bucket.

AWS allows block public access at two levels, account level and bucket level. Best
practise is to enable it at the account level so no bucket can accidentally expose data.
Block public access protects against public access control, public bucket policies,
existing bucket permissions and unintended cross-account access.

So it is essentially a master lock that overrides risky configurations. Here's a simple
way to think about it. Making an S3 bucket public without protection is like leaving your
house door unlocked because you trust everyone.

So block public access is like a building rule that says no doors can be left unlocked
even by mistake. In the real world, public access should be used only for things like
static website or public images. It should never be used for logs, backups, customer
data or database exports.

The key takeaway is simple. Security should be enforced by controls, not by
assumptions. With that, we'll move into S3 encryption options and understand how
data is protected even after access is controlled.
